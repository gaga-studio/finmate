# FinMate 기능추가 로드맵/판단 기준 v1.0

작성일: 2026-06-30  
용도: 기능 추가 회의에서 P0 범위가 흔들리지 않도록 판단 기준과 결정 이력을 고정한다.

## 1. 기능 분류

| 분류 | 정의 | 예시 |
| --- | --- | --- |
| Core | 또래 비교, 시뮬레이션, 미션 전환을 직접 강화 | 비교 지표 개선, 미션 추천 정확도, 홈 요약 |
| Social Impact | 청년이 금융을 작게 경험하고 학습하게 함 | 생일 위시머니, 퀴즈, 공식 교육 콘텐츠 |
| Engagement | 재방문, 포인트, 소셜, 공유 강화 | 포인트 지갑, 친구 행동 카드, 공동 미션 |
| Infra-Safety | 동의, 마스킹, audit log, 데이터/배포 안정성 | ConsentHistory, AuditLog, demo reset |

## 4. 기능 추가 회의 판단표

| 질문 | 답이 없으면 |
| --- | --- |
| Core, Social Impact, Engagement, Infra-Safety 중 어디에 속하는가? | 보류 |
| 어떤 사용자 문제를 해결하는가? | 보류 |
| P0 데모 흐름을 강화하는가, 흐리는가? | 흐리면 P1/P2 |
| 필요한 화면/API/테이블은 무엇인가? | 영향 범위 전까지 보류 |
| 개인정보/금융상품 권유 리스크가 있는가? | 표현 가이드 먼저 작성 |
| API가 몇 개 필요한가? | 3개 이상이면 복잡도 리뷰 후 P0 포함 여부 결정 |
| 7분 발표에서 10초 안에 설명 가능한가? | 설명 안 되면 로드맵 |
| Social Impact 지표와 연결되는가? | Social Impact 기능이라면 필수 |

## 7. P0 보호 원칙

- P0는 `또래 비교 → 시뮬레이션 → 미션 → 개인정보 안전장치`를 벗어나지 않는다.
- Birthday Wish Money는 중요하지만 P0 구현 기능이 아니다.
- LLM은 P0 필수 성공 조건이 아니다.
- 기능이 좋아 보여도 7분 발표 데모 흐름을 흐리면 P1/P2로 보낸다.
- “ESG”보다 “Social Impact” 표현을 우선 사용한다.
